# 2026-04-15 工作日志

## K线多周期 Bug 修复 + 数据库统一

### 问题根因
1. `klines_5min.date` 返回 "2020-01-02 09:35" 格式，Lightweight Charts 不支持（只支持 "YYYY-MM-DD" 或 Unix 秒数），导致图表渲染崩溃
2. 15m/1h/4h 周期没有聚合逻辑，直接显示 5 分钟原始数据
3. 两张表 (klines + klines_5min) 结构不统一，查询逻辑分散

### 修复方案

#### 数据库优化
- 创建统一表 `klines_all`，包含 `timeframe` 字段区分周期
- 迁移数据：2996 条日K + 69984 条 5min = 72980 条
- 索引：`(scenario_id, symbol, timeframe, dt)` 复合索引
- RLS：anon 只读策略

#### 前端修复 (supabase-loader.ts)
- 重写为统一从 `klines_all` 表查询
- 5min 数据的 `date` 字段改为 Unix 秒数（number），兼容 Lightweight Charts
- 新增 `aggregateMinuteKlines()` 函数：5m → 15m/1h/4h 前端聚合
- 分页加载逻辑保留（每页 1000 条）

#### 图表修复 (KLineChart.tsx)
- 新增 `toChartTime()` 工具函数：自动判断日期字符串 vs Unix 秒数
- 全部 `.date as Time` 替换为 `toChartTime(k.date)`
- 分钟级聚合（15m/1h/4h）由外部 SimulationPlayground 在传入前完成

#### 后端修复 (simulation.ts)
- 重写所有路由，从 TODO 占位变为真正从 Supabase 查询
- 新增 `/klines/:scenarioId/:symbol?timeframe=5m` 路由
- 安装 `@supabase/supabase-js` 到 server 包

## KLineChart 分钟K线排序崩溃 Bug 修复
- 根因：`markers.sort()` 用 `localeCompare()` 但 5min 数据的 time 是 `number` 类型（Unix 秒数），不是 string
- 同时修复：新闻标记在分钟模式下无法匹配（`"1577947500"` vs `"2020-01-20"`）
- 修复方案：
  1. `toChartTime()` 同时接受 `string | number` 输入
  2. 新增 `compareTime()` 函数兼容 number/string 排序
  3. 新增 `isMinuteData()` 判断当前是否分钟级数据
  4. 新增 `unixToDate()` 从 Unix 秒数提取日期
  5. 新闻匹配逻辑分分钟/日K两个分支处理
- 验证：两个场景页面 HTTP 200 + 零 lint 错误

## 分钟K线推进 + 自动播放控制栏
- Bug 根因：分钟K线一次性全量加载，没有按 currentDate 裁剪，导致点"下一天"数据不变
- 修复：新增 `clipMinuteKlines()` 函数按 currentDate 裁剪分钟数据
- 新增自动播放控制栏：
  - 播放/暂停按钮（▶ / ⏸）
  - 步长选择：5m / 1h / 1d
  - 速度选择：0.5x / 1x / 2x / 5x / 10x
  - 分钟播放索引追踪（minutePlayIndex）
  - 当天分钟数据播完自动跳到下一天
- 手动控制按钮："+1h"（推进1小时）和 "+1天"
- 顶部栏显示分钟进度信息（柱数/当前时间）
- 验证：三个场景页面全部 200 + 零 lint 错误
