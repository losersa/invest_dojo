## 1. 项目定位

### 1.1 一句话
InvestDojo 投资道场 —— **模拟炒股 × 量化回测 × 财报分析** 三合一平台，核心差异点是**人的决策与模型的判断在同一历史情境里并肩/对抗/协作**。

### 1.2 三大模块

| 模块 | 核心价值 | 状态 |
|------|---------|------|
| **模拟炒股** | 盲测/经典/自选 三种情景模拟 | v1 已上线 · v2 盲测重构中 |
| **量化模块** | 因子+训练+回测+推理+四种联动 | 设计完成，待落地 |
| **财报分析** | AI 辅助财报解读 | 规划中 |

### 1.3 关键非功能要求
- **Mac 单机能跑完整开发栈**（8GB 内存够用）
- **一份代码多端部署**（Web/iOS/Android/Desktop）
- **严格防未来函数**（`as_of` 机制贯穿全链路）

## 2. 整体架构分层

### 2.1 五层分层图

```
┌────────────────────────────────────────────────────────────────┐
│                         CLIENT 客户端层                         │
│  Web (Next.js) ┃ iOS/Android (Capacitor) ┃ Desktop (Tauri)      │
└──────────────────────────┬─────────────────────────────────────┘
                           │ HTTPS + WebSocket
┌──────────────────────────┴─────────────────────────────────────┐
│                  EDGE / GATEWAY 接入层                          │
│  Next.js Route Handlers (轻业务/鉴权/路由分发)                   │
└──────────────────────────┬─────────────────────────────────────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
┌──────────┴────────┐ ┌────┴──────────┐ ┌─┴──────────────────┐
│ ORCHESTRATION    │ │ BUSINESS SVC  │ │ COMPUTE SVC       │
│ 会话编排层         │ │ 业务服务       │ │ Python 计算服务    │
│                  │ │               │ │                   │
│ Session          │ │ User / Auth   │ │ feature-svc       │
│ Orchestrator     │ │ Model Market  │ │ train-svc         │
│ (Node 长连接)     │ │ Marketplace   │ │ infer-svc         │
│                  │ │ (Next.js)     │ │ backtest-svc      │
│                  │ │               │ │ monitor-svc       │
└──────────┬───────┘ └──────┬────────┘ └─────────┬─────────┘
           │                │                    │
           └────────────────┼────────────────────┘
                            │
┌───────────────────────────┴─────────────────────────────────────┐
│                      DATA 数据层                                 │
│  Supabase (Postgres)  ┃  Redis  ┃  MinIO / S3  ┃  消息队列      │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 每层职责

**CLIENT 客户端**
- Next.js SSR/CSR 混合渲染
- Capacitor 打包为 iOS/Android 原生壳
- Tauri 打包为 macOS/Windows 桌面应用
- 统一使用同一份前端代码

**EDGE / GATEWAY 接入层**
- Next.js Route Handlers 做轻量业务和鉴权
- 不做重计算（转发到 compute 服务）
- 承载用户/订阅/市场等 CRUD
- 规模：2~3 个实例即可

**ORCHESTRATION 会话编排层**
- **InvestDojo 的心脏**：让人和模型在同一时钟下看到同一份真相
- 长连接（WebSocket/SSE）维护会话状态
- 跨调用数据服务和推理服务
- 详见 [04_联动机制](./04_联动机制.md)

**BUSINESS SVC 业务服务**
- 承载纯业务逻辑（CRUD/市场/订阅/权限）
- 与 ORCHESTRATION 并列，不交叉职责

**COMPUTE SVC Python 计算服务**
- 所有重计算放在这里
- FastAPI + Ray Serve + Celery
- 通过 HTTP/gRPC 对外暴露

**DATA 数据层**
- Supabase 做主库
- Redis 做热缓存
- MinIO 存模型文件/Notebook/报告
- 消息队列用 Redis Pub/Sub（规模上来再换 NATS）

### 2.3 分工原则

| 工作类型 | 归属 |
|---------|------|
| UI 渲染 / 交互 | CLIENT |
| 鉴权 / 路由 / 轻业务 | EDGE |
| 长连接 / 时钟 / 会话状态 | ORCHESTRATION |
| CRUD / 用户态业务 | BUSINESS |
| 训练 / 推理 / 回测 / 特征计算 | COMPUTE |
| 数据持久化 | DATA |

**核心原则**：
- Node 做编排，Python 做计算
- 长连接统一走 ORCHESTRATION 层
- 重计算**不允许**放在 EDGE / BUSINESS 层

## 5. 关键数据流

### 5.1 因子计算流（夜间批处理）

```
17:00 每日收盘
    ↓
Celery Beat 触发
    ↓
feature-svc · batch_compute_daily
    ↓
取当天 K 线 + 财报（增量）
    ↓
遍历所有预计算因子（≈200 个）
    ↓
写入 feature_values 表
    ↓
Redis Pub/Sub 发 "factor_updated"
    ↓
infer-svc 收到事件 → 刷新缓存
```

### 5.2 模型训练流

```
用户点"开始训练"
    ↓
Next.js → train-svc POST /train
    ↓
train-svc 创建 models 记录（status=training）
    ↓
Celery 入队训练任务
    ↓
Worker 拉取 → 取数据 → 计算特征 → 拟合 → 评估 → 保存到 MinIO
    ↓
更新 models 记录（status=ready + metrics）
    ↓
WebSocket 推送给前端 "training_completed"
    ↓
用户查看报告 → 一键回测
```

### 5.3 联动模式数据流（副驾模式示例）

```
用户进入盲测
    ↓
Next.js → POST /api/v1/sessions (mode=copilot_observer)
    ↓
session-orch 创建 Session + 附加模型订阅
    ↓
返回 WebSocket URL
    ↓
前端建立 WS 连接
    ↓
━━━ 用户点"下一天" ━━━
    ↓
WS 发 { type: "request_tick" }
    ↓
session-orch:
  ├── 推进时钟（current_ts += 1 day）
  ├── 调 data-svc 取新 K 线 + 新闻
  ├── 调 infer-svc 传 as_of → 拿 Signal[]
  └── WS 推送给前端 { type: "tick", payload: { new_data, signals }}
    ↓
前端更新 K 线 + 副驾面板
    ↓
（严格防未来函数：所有调用都强制 as_of <= session.clock.current_ts）
```

### 5.4 数据闭环流（模式 ④）

```
用户每次下单
    ↓
session-orch 记录 order
    ↓
异步计算标签（需要等 N 天后的真实收益）
    ↓
N 天后 training_samples 表获得完整样本
    ↓
（用户主动）选择贡献到公共池
    ↓
匿名化处理 → shared_samples 表
    ↓
供未来训练"模仿用户模型" / "纠错模型" / "群体智慧模型"
```

## 6. 跨模块依赖关系

### 6.1 依赖矩阵

|          | 依赖 data | 依赖 feature | 依赖 model | 依赖 infer | 依赖 session |
|----------|:---------:|:------------:|:----------:|:----------:|:------------:|
| 模拟炒股 UI | ✓ | - | - | - | ✓ |
| 因子库 UI  | ✓ | ✓ | - | - | - |
| 训练 UI   | ✓ | ✓ | ✓ | - | - |
| 回测 UI   | ✓ | ✓ | ✓ | ✓ | - |
| 联动 UI   | ✓ | ✓ | ✓ | ✓ | ✓ |

### 6.2 关键耦合点

**Session Orchestrator ↔ infer-svc**
- Session 推进时钟 → 触发 infer
- infer 结果回传给 Session
- **永远不能反向调用**（避免循环）

**feature-svc ↔ data**
- feature 依赖 data 取原始数据
- feature 把算好的因子值写回 `feature_values` 表
- 其他服务直接查 `feature_values`，不重新算

**train-svc ↔ infer-svc**
- train 训好的模型文件落到 MinIO
- infer 按需从 MinIO 加载
- 两者通过 `models` 表协调状态

### 6.3 启动顺序

```
data 层就绪
   ↓
feature-svc 可工作（独立，无其他依赖）
   ↓
train-svc 可工作（依赖 feature-svc 取数据）
   ↓
infer-svc 可工作（依赖 MinIO 加载模型）
   ↓
backtest-svc 可工作（依赖 data + infer）
   ↓
session-orch 可工作（依赖 data + infer）
   ↓
前端/EDGE 可工作
```

## 7. 安全与权限

### 7.1 认证

- 统一使用 **Supabase JWT**
- Access Token 过期时间：1 小时
- Refresh Token：7 天
- 所有后端服务都能解析并校验 JWT（通过 Supabase JWKS）

### 7.2 授权（RLS）

Supabase Row Level Security 策略：

```sql
-- 用户只能访问自己的模型
CREATE POLICY "models_owner_only" ON models
    FOR ALL USING (owner = current_user_id());

-- 公开模型任何人可读
CREATE POLICY "models_public_read" ON models
    FOR SELECT USING (visibility = 'public');

-- 官方模型只有 service_role 能写
CREATE POLICY "models_platform_write" ON models
    FOR INSERT WITH CHECK (
        owner != 'platform' OR current_role() = 'service_role'
    );
```

### 7.3 用户模型隔离

用户上传的模型在 **gVisor 沙箱**运行：
- 不允许网络访问
- 不允许文件系统访问（除了平台提供的数据 API）
- CPU 配额：1 核
- 内存配额：1GB
- 执行超时：5 秒

### 7.4 API 速率限制

见 [API 约定 §6](../api/00_约定.md#6-速率限制)。

### 7.5 数据隐私

- 用户样本默认私有
- 贡献公共池需显式同意 + 严格匿名化
- GDPR 式数据主权（一键删除）

## 8. 观测性

### 8.1 指标（Metrics）

Prometheus 采集：

| 类别 | 指标 | 告警阈值 |
|------|------|---------|
| 服务健康 | up / request_rate / error_rate | error_rate > 1% |
| 推理 | inference_latency_p99 | > 500ms |
| 训练 | training_queue_depth | > 20 |
| 会话 | active_sessions | - |
| DB | query_latency_p99 | > 100ms |

### 8.2 日志（Logs）

- 统一 JSON 格式
- 每个 request 带 `request_id`（贯穿所有下游调用）
- 结构化字段：`level` / `service` / `user_id` / `session_id` / `model_id`
- Loki 集中查询

### 8.3 追踪（Traces）

OpenTelemetry 埋点关键路径：
- 前端请求 → EDGE → 各服务 → DB
- WebSocket 消息流
- Celery 任务全链路

### 8.4 用户行为埋点

埋点事件（产品分析用）：
- `session_started` / `session_finished`
- `model_trained` / `model_used`
- `backtest_run`
- `suggestion_accepted` / `suggestion_rejected`

## 10. 风险清单

| 风险 | 影响 | 缓解 |
|------|------|------|
| Mac 本地跑不动 | 开发体验差 | 关键服务可拆到远程开发容器 |

| BaoStock 数据源不稳 | 数据采集失败 | 多源互补（AKShare/Tushare） |
| 联动模式延迟高 | 用户体验差 | 预计算 + 流式推理 + 模型缓存 |
| 用户上传恶意模型 | 系统被攻击 | gVisor 沙箱 + 白名单依赖 |
| 训练任务阻塞 | 用户等太久 | 队列隔离（免费/付费）+ 超时保护 |
