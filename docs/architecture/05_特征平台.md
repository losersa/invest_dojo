## 定位

**特征平台** = 让用户在页面上写公式 → 自动计算 → 自动调度 → 自动上线 → 可订阅。

等价于：InvestDojo 的"量化因子 UGC 生产工厂"。

不是什么：
- 不是一个"因子库文档网站"（那是被动浏览）
- 不是"回测系统"（特征平台产出的是因子值，下游才是回测）
- 不是通用"特征工程工具"（聚焦金融因子，不做 ML 通用 pipeline）

---

## 五层架构栈

### ① 定义层（Definition）

**目的**：用户怎么描述因子，越易读越好。

**当前**：
- DSL 解析器 `feature-svc/factors/dsl_parser.py`（Pratt 法）
- 支持 15 函数：MA / EMA / STD / MAX / MIN / DIFF / PCT / RANK / RSI / MACD / BOLL / KDJ / cross_up / cross_down / ARGMAX
- 支持 28 内置字段：K 线 (9) + 基本面 (16) + 衍生 (3)
- AST 支持序列化，给前端可视化用
- 200+ 内置因子（technical 80 / valuation 40 / growth 40 / sentiment 40）

**v1 要补**：
- 前端 Monaco 编辑器 + DSL 语法高亮
- 公式模板库（"MA 金叉"、"RSI 超卖反弹"等一键插入）
- Python 原生公式（沙箱模式，给高级用户）

---

### ② 调度层（Scheduling）

**目的**：什么时候算，按什么顺序。

**当前**：
- Celery Beat 每日 17:00 全局触发（`train-svc/celery_worker.py`）
- 手动 REST：`POST /factors/compute`
- 任务：`feature.compute_incremental` / `feature.compute_range`

**v1 要补**：
- 每个因子可独立配置 cron（存 `factor_definitions.schedule` 字段）
- Beat 动态加载：改了调度不用重启 worker
- 优先级队列：高优先级因子插队

**v2 愿景**：
- 事件触发：有新公告 / 异动事件 → 触发相关因子
- Feature DAG：因子 A 依赖因子 B，自动 topo 排序

---

### ③ 计算层（Computation）

**目的**：拿到定义和触发信号后真实算。

**当前**：
- pandas 向量化引擎 `factors/engine.py`
- batch 模式：多股票一次 panel 共享 I/O
- 吞吐：3000 股 × 252 天 单因子 25~60ms
- 批量脚本：`scripts/backfill_factors.py` + `factors/batch_compute.py`
- 实测：100 股 × 204 因子 × 1 天 = 14 秒（含写入 20K 条）

**v1 保持**，不做改动。

**v2 愿景**：
- 流式计算（Kafka/NATS 订阅 Tick，日内实时算）
- 分布式（Ray / Dask），单机不够时扩展
- 增量计算（只算 diff，不重算整段）

---

### ④ 存储层（Storage）

**目的**：算完放哪，既支持历史回溯又支持在线快查。

**当前**：
- PostgreSQL `feature_values` 表 + 按年分区（2014-2024）
- Schema：`(factor_id, symbol, date, value_num, value_bool, computed_at)`
- upsert 幂等（`on_conflict=factor_id,symbol,date`）
- 查询延迟 ~200ms

**v1 要补**：
- Redis 在线缓存：最近一天的因子值，< 10ms
- 失效策略：每天 17:30 自动刷新

**v2 愿景**：
- Point-in-Time 读取：`get_features(symbol, as_of="2024-06-15")` 返回当时可见的因子值（回测防未来函数）
- 冷数据归档到 Parquet / S3

---

### ⑤ 服务层（Serving）

**目的**：业务方（前端 / 策略 / 回测 / 模型）怎么消费。

**当前**：
- REST 3 个：`GET /factors`（列表）、`POST /factors/validate`、`POST /factors/compute`
- TypeScript SDK（`packages/api/`）

**v1 要补**：
- 前端页：因子库浏览、详情、编辑器
- WebSocket 订阅：因子触发时推送通知
- 完整 CRUD：POST/PUT/DELETE/compare 等（T-3.06）

**v2 愿景**：
- GraphQL 批查（一次查 100 个因子）
- Python notebook SDK（研究员在 Jupyter 里用）
- 订阅 → 微信 / 邮件通知

## 数据流（端到端）

```
用户        校验预览          保存调度          批量计算         落库+缓存
 ↓            ↓                 ↓                ↓                ↓
网页  →  /factors/      →  factor_      →   batch_      →  feature_values
编辑器     validate          definitions     compute        + Redis
写公式    解析 AST + mock   插行 + 加 Beat   向量化跑     PG 永久 + Redis 热数据
```

---

## 关键设计选择（踩坑后固化）

| 决策 | 为啥这么选 |
|---|---|
| DSL 用 Pratt 而不是 PEG/ANTLR | 表达式语法简单，无外部依赖，出错定位好 |
| fundamentals 展平为内置字段而不是 `fundamental("roeAvg")` 函数 | YAML 可读性压倒一切，通用性靠函数补 |
| announce_date + 1 天前向填充 | 防未来函数的最基本手段，公告当天还在消化中 |
| PostgreSQL + 分区表（不上专用时序库 InfluxDB） | 避免引入新运维负担，分区已够用到 2030 |
| feature_values long 表（不宽表） | 因子数量会涨到千级，宽表加列成本高 |
| Celery 复用 train-worker（不起独立 feature-worker） | Sprint 0 省运维 |

## 未来扩展点（v2 / v3 留位置）

**v2（企业级）**：
- 接多数据源（新闻 / 龙虎榜 / 北向 / 情绪）
- 实时流计算
- Point-in-Time 读取
- 自动 backfill 调度器
- 数据质量规则 + A/B 分桶

**v3（生态型）**：
- Marketplace（发布/订阅/打赏）
- 多租户 SaaS（对外输出）
- 因子 IDE（项目化 + Git）
- 联邦学习
