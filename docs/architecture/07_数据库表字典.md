# 📊 基础数据

## `symbols` — 股票元数据

> **一句话**：全市场 A 股的"身份证"（代码 / 名称 / 行业 / 上市日）。

**用途**
- 前端选股下拉、搜索
- 回测前校验股票是否上市
- factor 计算里反向查"这个 code 属于哪个行业"

**关键字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | TEXT PK | 6 位代码，如 `600519` |
| `market` | TEXT | `SH` / `SZ` / `BJ` |
| `name` | TEXT | 中文全称，如 "贵州茅台" |
| `short_name` | TEXT | 简称 |
| `industry` | TEXT | 一级行业，如 "白酒" |
| `industry_level2` | TEXT | 二级行业，如 "食品饮料" |
| `listed_at` / `delisted_at` | DATE | 上市/退市日 |
| `total_share` / `float_share` | NUMERIC | 总/流通股本（亿股） |
| `status` | TEXT | `normal` / `suspended` / `delisted` |
| `tags` | JSONB | 指数成分，如 `["沪深300", "红利"]` |

**谁读谁写**
- 写：`scripts/seed_data.py`（AKShare 日更）
- 读：全服务，用 `client.select("symbols", filters={"code": f"in.({...})"})`

**当前数据量**：**5,524 行**

**典型查询**
```sql
-- 沪深300 成分股
SELECT code, name FROM symbols WHERE tags @> '["沪深300"]';
```

## `klines_all` — K 线统一表

> **一句话**：**整个系统最核心的大表**。同时承载"全市场日 K"和"场景切片 5m K"两种数据。

**设计亮点**
- 一张表搞定所有 timeframe（`1d` / `5m` / `1h` ...），不再分 `klines_daily` / `klines_5min`
- `scenario_id` 字段**可为 NULL**：`NULL` 表示全市场数据，非 NULL 表示场景切片
- `adj_factor` 支持前/后复权切换（目前默认 `1` = 不复权）
- 唯一约束：`(symbol, timeframe, dt)` 防止重复插入

**关键字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `scenario_id` | TEXT NULL | `NULL` = 全市场；非空 = 场景切片 |
| `symbol` | TEXT | 股票代码 |
| `timeframe` | TEXT | `1d` / `5m` / `1h` / ... |
| `dt` | TIMESTAMPTZ | **北京时间存为 UTC**，查时要 `AT TIME ZONE 'Asia/Shanghai'` |
| `open` / `high` / `low` / `close` | NUMERIC | OHLC |
| `volume` | NUMERIC | 成交量（股） |
| `turnover` | NUMERIC | 成交额（元） |
| `pre_close` | NUMERIC | 昨收 |
| `change_amount` / `change_percent` | NUMERIC | 涨跌额/幅 |
| `adj_factor` | NUMERIC | 复权因子 |

**谁读谁写**
- 写：
  - `scripts/seed_data.py`（AKShare 日K 全市场）
  - `scripts/seed_baostock.py`（BaoStock 5m + 历史日K）
  - `scripts/update_daily_klines.py`（launchd 每天 19:00 增量）
- 读：
  - data-svc `/api/v1/data/klines`（前端 K 线图）
  - feature-svc `panel_loader`（因子计算底数据）
  - 模拟炒股前端（`supabase-loader.ts`）

**当前数据量**：**5,941,010 行**（4,376 支股票 × 2020-01-02 ~ 2026-04-28）
- 场景切片日 K：2,253 行（4 个场景）
- 场景切片 5m K：108,144 行

**踩坑警报** ⚠️
1. **timestamptz 查询边界**：`dt.lte.{end}` 会漏掉北京时间当日最后几小时（UTC 在第二天）。用 `dt.lt.{end+1 天}` 替代
2. **PostgREST 硬限制**：单次 `select` 最多 1000 行，大查询必须分页（`client.select_all` 已封装）
3. `scenario_id IS NULL` 时的 UNIQUE 约束依赖 partial index，手写 SQL 注意

**典型查询**
```python
# 取茅台 2024 年日 K
rows = client.select_all(
    "klines_all",
    columns="dt,open,high,low,close,volume",
    filters={
        "symbol": "eq.600519",
        "timeframe": "eq.1d",
        "scenario_id": "is.null",
        "and": "(dt.gte.2024-01-01,dt.lt.2025-01-01)",
    },
    order="dt.asc",
)
```

## `fundamentals` — 财报

> **一句话**：三大报表（利润表/资产负债表/现金流）的季度/半年/年度数据。**防未来函数的关键**。

**字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `symbol` | TEXT | 股票代码 |
| `report_date` | TEXT | 报告期，如 `2024-Q1` / `2024-H1` / `2024-Q3` / `2024` |
| `announce_date` | **DATE** | **公告日** —— 防未来函数的命根子 |
| `statement` | TEXT | `income` / `balance` / `cashflow` |
| `data` | JSONB | 原始字段（`revenue` / `net_profit` / ...） |
| `derived` | JSONB | 派生指标（`roe` / `eps` / ...） |

**UNIQUE**: `(symbol, report_date, statement)`

**为什么 announce_date 这么重要**
财报发布存在**滞后**：`2024-Q1` 报表一般在 4 月底~5 月底才公告。
因子计算时，panel_loader 会把 `announce_date + 1 天`之后才让该报表对 DSL 可见，
避免"当前 bar 知道还没公告的财报"的未来函数泄露。

**谁读谁写**
- 写：`scripts/seed_fundamentals.py`（BaoStock，T-1.05 跑了两轮共 13 小时）
- 读：feature-svc 的 `panel_loader._load_fundamentals()`，展平为 DSL 可用字段（`eps_ttm` / `roe` / `yoy_ni` 等）

**当前数据量**：**559,122 行**，覆盖 5,176 支股票

**踩坑**
- **BaoStock 凌晨 23:30~00:30 维护断连** → 多轮断点续传兜底
- **美国 IP 访问被屏蔽** → GitHub Actions 跑不通，必须本地跑

# 🧮 量化层

## `factor_definitions` — 因子定义

> **一句话**：因子库的"目录"，每条记录是一个可被计算的因子公式。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | TEXT PK | 如 `ma_cross_20_60`、`custom_abcdef_12345678` |
| `name` / `name_en` | TEXT | 中英名 |
| `description` / `long_description` | TEXT | 短描述 / Markdown 长文 |
| `category` | TEXT | `technical` / `valuation` / `growth` / `sentiment` / `fundamental` / `macro` / `custom` |
| `tags` | JSONB | `["均线", "趋势"]` |
| `formula` | TEXT | DSL 公式，如 `MA(close, 20) cross_up MA(close, 60)` |
| `formula_type` | TEXT | `dsl` / `python` |
| `output_type` | TEXT | `boolean` / `scalar` / `rank` |
| `output_range` | JSONB | 值域 `[min, max]` |
| `lookback_days` | INT | 需要的历史天数 |
| `update_frequency` | TEXT | `daily` / `realtime` / `hourly` |
| `version` | INT | 改公式时自动 +1 |
| `owner` | TEXT | `platform` 或 user_id |
| `visibility` | TEXT | `public` / `private` / `unlisted` |
| `stats_cache` | JSONB | 预聚合的表现统计 |
| `deprecated_at` | TIMESTAMPTZ | 软删标记 |

**谁读谁写**
- 写：
  - `scripts/seed_factors.py`（平台因子 YAML → DB）
  - feature-svc `POST /factors`（用户新建）
  - feature-svc `PUT /factors/{id}`（改公式自动 version+1）
- 读：feature-svc 列表 / 详情 / 计算 API

**当前数据量**：**205 个 platform 因子**
- 技术类 84（含 80 个 T-3.03 标准因子 + 4 示范）
- 估值类 40
- 成长类 40
- 情绪类 40
- 宏观类 0（v1 后扩）

**关键决策**：平台官方因子和用户因子**共表**，只靠 `owner='platform'` 区分（ADR-0002）。

## `feature_values` — 因子值矩阵

> **一句话**：**系统里数据量最大的表**（预计 15 亿行）。存所有因子在所有股票所有日期的计算值。

**结构**
- 按年分区（17 张分区表 `feature_values_2014` ~ `feature_values_2030`）
- 主键：`(factor_id, symbol, date)`

| 字段 | 类型 | 说明 |
|------|------|------|
| `factor_id` | TEXT | |
| `symbol` | TEXT | |
| `date` | DATE | |
| `value_num` | DOUBLE PRECISION | scalar/rank 结果 |
| `value_bool` | BOOLEAN | boolean 结果 |
| `computed_at` | TIMESTAMPTZ | 计算时间戳 |

只用一个字段（`value_num` 或 `value_bool`），另一个为 NULL。

**谁读谁写**
- 写：
  - feature-svc `batch_compute.compute_and_save`（T-3.05 实时计算）
  - `scripts/backfill_factors.py`（历史回填）
  - train-svc Celery 每日 17:00 增量（Beat schedule）
- 读：
  - feature-svc `/factors/{id}/history`（历史值查询）
  - feature-svc `/factors/{id}/performance`（聚合统计）
  - feature-svc `/factors/batch-query`（多因子多股快照）

**当前数据量**：约 **2,000+ 万行**（T-3.05 backfill 中，预计 2024 Q4 跑完 1,600 万）

**踩坑**
- 批量 upsert 用 PostgREST POST（~1600 行/s）比 Management API SQL 快 3-4x
- 大范围查询必须带 `date` 区间，让分区裁剪生效

## `models` — 模型注册表

> **一句话**：所有模型（用户训练的 + 平台官方的 + 用户上传的）统一注册地。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | TEXT PK | `model_abc123` |
| `name` / `version` | TEXT | 同 owner 下 UNIQUE |
| `owner` | TEXT | user_id 或 `platform` |
| `source` | TEXT | `internal`（平台训的） / `uploaded`（用户上传 pkl） / `external_signal` / `official` |
| `algorithm` | TEXT | LightGBM / XGBoost / LSTM ... |
| `input_features` | JSONB | 依赖的 factor_id 数组 |
| `target` | TEXT | 预测目标 |
| `output_type` | TEXT | `classification` / `regression` / `ranking` |
| `training_range` | JSONB | `{start, end}` 训练区间 |
| `training_samples` | INT | 样本数 |
| `validation_metrics` | JSONB | `{accuracy, sharpe, ...}` |
| `file_path` | TEXT | MinIO 路径 |
| `visibility` | TEXT | `public` / `private` |
| `metadata` | JSONB | 官方模型在此存 PlatformModelExtras（tier/category/retrain_schedule） |
| `status` | TEXT | `draft` / `training` / `ready` / `failed` / `deprecated` |
| `usage_count` / `last_used_at` | | 热度统计 |

**关键约束**：`UNIQUE (owner, name, version)`

**ADR 对应**：官方模型走和用户模型一样的注册路径，只是 `owner='platform'`（详见 ADR-0002）。

## `model_versions` — 模型版本

> **一句话**：一个 model 多个版本，只有一个 `is_current = TRUE`。

| 字段 | 说明 |
|------|------|
| `(model_id, version)` | 联合主键 |
| `file_path` | MinIO 里该版本的文件 |
| `validation_metrics` | 该版本的评测结果 |
| `is_current` | 当前生效版本 |
| `training_job_id` | 关联到 `training_jobs` |

UNIQUE 索引 `(model_id) WHERE is_current = TRUE` 保证只有一个当前版。

# 🎮 联动会话

## `sessions` — 联动会话

> **一句话**：四种模式（solo / copilot_observer / pk / copilot_interactive）的会话核心。

| 字段 | 说明 |
|------|------|
| `id` | PK |
| `user_id` / `scenario_id` | 谁在哪个场景 |
| `mode` | 四种模式之一 |
| `blind_options` | JSONB 盲测开关（隐藏股票名/日期/行业等） |
| `clock_start_ts` / `clock_end_ts` / `clock_current_ts` | **会话时钟**（历史时间，不是墙钟） |
| `tick_granularity` | `5m` / `1d` |
| `total_ticks` / `current_tick` | 进度 |
| `initial_capital` | 初始资金 |
| `status` | `initializing` / `running` / `paused` / `finished` / `abandoned` |
| `summary` | JSONB 结束时的对账数据 |

**核心设计**：会话内的所有操作（下单、AI 信号、推进 tick）都通过 `session_events` 记录。

---

## `session_participants` — 会话参与者

> **一句话**：PK 模式里同一 session 会有多个参与者（人 + 1~N 个模型）。

| 字段 | 说明 |
|------|------|
| `participant_type` | `human` / `model` |
| `participant_id` | user_id 或 model_id |
| `cash` / `total_value` / `total_return` / `max_drawdown` | 实时账户状态 |
| `stats` | JSONB 细节（交易次数、胜率等） |

---

## `session_events` — 会话事件日志

> **一句话**：会话的"黑匣子"，每个 tick / 下单 / 模型信号都落一条。

| 字段 | 说明 |
|------|------|
| `ts` | 会话时钟（历史时间） |
| `real_ts` | 墙钟（真实产生时间） |
| `actor_type` / `actor_id` | 谁触发的 |
| `event_type` | `tick_advance` / `order_placed` / `model_signal` / ... |
| `payload` | JSONB 事件详情 |
| `sequence` | 本会话内严格递增序号 |

---

## `orders` — 订单

> **一句话**：会话内所有订单，人和模型共用。

| 字段 | 说明 |
|------|------|
| `session_id` / `participant_id` | 谁在哪个会话下的 |
| `ts` | 会话时钟 |
| `symbol` / `side` / `quantity` / `price` / `order_type` | 标准订单字段 |
| `status` | `filled` / `partial` / `rejected` / `pending` |
| `commission` / `stamp_tax` / `slippage_cost` | 成本明细 |
| `triggered_by` / `signal_id` | 关联到 AI 信号（如 `model_suggestion:xxx`） |

---

## `training_samples` — 训练样本

> **一句话**：数据闭环 —— 把"人在会话里做的决策 + N 天后的未来收益"变成训练样本。

| 字段 | 说明 |
|------|------|
| `features` | JSONB 决策时刻的特征快照 |
| `label` | `buy_heavy` / `buy_light` / `hold` / `sell_light` / `sell_heavy` |
| `context` | JSONB 市场状态 + future_return_N |
| `label_resolved_at` | 标签回填时间（需等 N 日后才能算未来收益） |
| `shared_to_public` | 用户授权贡献到公共池 |

**决策来源**：仅当 `user_preferences.data_collection_opt_in = TRUE` 才采集。

# 📐 架构规范

## 命名约定
- 表名：下划线、复数、小写（`feature_values` ✅）
- 字段：下划线、单数
- 时间字段：`_at` 表墙钟 / `_ts` 表会话时钟 / `date` 表日期 / `dt` 表带时区时间戳

## 分区策略
| 表 | 策略 | 原因 |
|----|------|------|
| `feature_values` | **按年 RANGE** | 查询必带 date 区间，分区裁剪 |
| `klines_all` | 单表 | 5941 万行还撑得住，Phase 3 再考虑 |
| `session_events` | 单表（暂） | 大量后按 `session_id` hash 分区 |

## JSONB 使用边界
✅ **适合 JSONB**
- 异构配置（`backtests.config`、`scenarios.tags`）
- 样本快照（`training_samples.features`）
- 枚举集合（`symbols.tags`）

❌ **不适合 JSONB**
- 需要频繁 `GROUP BY` / `SORT BY` 的字段 → 拆成独立列
- `fundamentals.data` 里被因子频繁引用的字段 → panel_loader 展平为 DSL 字段（ADR 方案 A）

## 时间处理铁律

1. **存储：UTC**。Supabase `timestamptz` 存北京时间**必须先转 UTC**（`04-28 → 04-27T16:00`）
2. **查询：边界 +1**。`dt.lte.{end}` 漏数据，用 `dt.lt.{end+1}`
3. **展示：`AT TIME ZONE 'Asia/Shanghai'`** 转回北京时间
4. **pandas DatetimeIndex 带时区**时，切片必须用 `pd.Timestamp(start, tz=idx.tz)` 对齐

## 防未来函数

| 场景 | 机制 |
|------|------|
| 财报 | `announce_date + 1 天` 之后才对 DSL 可见 |
| 新闻 | `published_at < as_of` 硬过滤 |
| K 线 | `dt < as_of` 硬过滤（data-svc 强制注入） |
| 回测 | `common/as_of_enforcer.py` 契约测试兜底 |
