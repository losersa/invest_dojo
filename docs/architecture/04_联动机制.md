## 1. 定位

### 1.1 一句话

**Session Orchestrator（SO）是联动模式的心脏**。它解决 InvestDojo 最核心的技术挑战：

> 让人和模型在同一时刻看到同一份数据，做出独立决策，并被公平撮合。

### 1.2 为什么需要独立的编排层

如果把会话逻辑分散到前端或业务服务里：

- **时钟漂移**：人的操作和模型推理在不同时钟坐标下，对账失真
- **数据泄漏**：任一参与者不小心看到未来数据，整个实验失效
- **调度混乱**：时钟推进 / 数据广播 / 撮合 / 信号推送必须严格有序

SO 独立成层后：
- 时钟单调性由 SO 强制
- 所有数据查询经 SO 注入 `as_of`
- 参与者（人 / 模型）都订阅 SO 的广播

### 1.3 职责清单

| 职责 | 是否 |
|------|:---:|
| 会话状态管理（创建/暂停/恢复/结束） | ✅ |
| 时钟推进（严格单调） | ✅ |
| 数据广播（每个 tick 推给所有订阅者） | ✅ |
| 模型推理协调（调 infer-svc） | ✅ |
| 撮合引擎（应用滑点/费率/T+1） | ✅ |
| 事件记录（写 session_events） | ✅ |
| 防未来函数（强制 as_of 注入） | ✅ |
| 结束对账（生成 debrief） | ✅ |
| **推理计算** | ❌（走 infer-svc） |
| **因子计算** | ❌（走 feature-svc） |
| **用户/模型 CRUD** | ❌（走 EDGE 层） |
| **回测** | ❌（走 backtest-svc） |

## 2. 核心概念

### 2.1 Session

一次完整的模拟交易，从创建到结束全程由同一 SO 实例持有。

```typescript
interface Session {
  id: string;
  mode: SessionMode;
  clock: Clock;
  participants: Participant[];
  portfolios: Map<ParticipantId, Portfolio>;
  order_book: OrderBook;
  status: 'initializing' | 'running' | 'paused' | 'finished' | 'abandoned';
  blind_options?: BlindOptions;
  data_collection_opt_in: boolean;
  created_at: string;
}
```

### 2.2 Clock（时钟）

```typescript
interface Clock {
  start_ts: string;            // 场景开始（历史时间）
  end_ts: string;              // 场景结束
  current_ts: string;          // 当前模拟时间（关键）
  granularity: '5m' | '1h' | '1d';
  total_ticks: number;
  current_tick: number;
}
```

**关键特性**：
- `current_ts` **严格单调递增**，不可回退
- 所有数据查询都以 `current_ts` 作为 `as_of`
- 人操作的墙钟和会话的模拟钟**互不相关**

### 2.3 Participant（参与者）

三种角色：

```typescript
type Participant =
  | { type: 'human'; id: string; display_name: string; }
  | { type: 'model'; id: string; model_version: string; decision_timeout_seconds: number; }
  | { type: 'observer'; id: string; }          // 只看信号不下单
```

### 2.4 Tick 事件

每次推进是原子操作，触发级联反应：

```
Tick
  ├─ 1. 时钟推进  current_ts += granularity
  ├─ 2. 查询新数据（K 线、新闻）
  ├─ 3. 按模式决定是否请求模型推理
  ├─ 4. 广播给订阅者
  ├─ 5. 等待参与者响应（下单 / 决策）
  ├─ 6. 撮合订单
  ├─ 7. 更新 portfolios
  ├─ 8. 记录 session_events
  └─ 9. 返回 tick 结果
```

### 2.5 Signal

模型输出，详见 [推理 API §2.1](../api/05_推理API.md#21-signal统一信号格式)。

## 3. Session Orchestrator 架构

### 3.1 内部组件

```
┌─────────────────────────────────────────────────────────┐
│             Session Orchestrator (Node 22)              │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │  WebSocket Gateway                                │  │
│  │  · 连接管理（per-session room）                     │  │
│  │  · 心跳 · 消息路由 · JWT 鉴权                       │  │
│  └──────────────────────┬────────────────────────────┘  │
│                         │                               │
│  ┌──────────────────────┴────────────────────────────┐  │
│  │  Session Manager                                  │  │
│  │  · 状态机 · 时钟推进 · 事件总线                     │  │
│  └──────────────────────┬────────────────────────────┘  │
│                         │                               │
│  ┌─────┬────────┬───────┼────────┬──────────┬─────────┐ │
│  ↓     ↓        ↓       ↓        ↓          ↓         │ │
│ Broad Match   Infer  AsOf     Event      Recovery     │ │
│ cast  Engine  Client Enforcer Recorder   Manager      │ │
└────────────────────────────────────────────────────────┘
```

| 组件 | 职责 |
|------|------|
| **WebSocket Gateway** | 连接管理、消息路由、心跳、鉴权 |
| **Session Manager** | 状态机、时钟推进、协调各组件 |
| **Broadcast Service** | 把 tick 广播给订阅者，保证顺序 |
| **Match Engine** | 订单撮合，应用滑点/费率/T+1 |
| **Infer Client** | 向 infer-svc 请求信号，带超时 |
| **AsOf Enforcer** | 拦截绕过 `as_of` 的调用 |
| **Event Recorder** | 批量异步写 session_events |
| **Recovery Manager** | 断电恢复、状态重建 |

### 3.2 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 运行时 | Node 22 | 与 Next.js 统一 |
| WebSocket 库 | `ws` + 自建 room | 避免 socket.io 冗余 |
| 热状态存储 | Redis | 毫秒级读写 |
| 持久化 | Postgres (Supabase) | 会话结束落库 |
| 跨实例协调 | Redis Pub/Sub | 轻量，够用 |
| HTTP 客户端 | `undici` | 快 |
| 调度 | 事件循环 + setImmediate | 无需额外框架 |

## 4. 会话生命周期

### 4.1 状态机

```
       ┌─────────────┐
       │initializing │
       └──────┬──────┘
              │ start()
              ↓
       ┌─────────────┐
       │   running   │←──────┐
       └──────┬──────┘       │
              │ pause()      │ resume()
              ↓              │
       ┌─────────────┐───────┘
       │   paused    │
       └──────┬──────┘
              │ finish() / timeout / abandon()
              ↓
       ┌─────────────┐
       │  finished   │ （终态）
       └─────────────┘
```

### 4.2 合法转换

| 来源 | 目标 | 触发 |
|------|------|------|
| initializing | running | `POST /sessions/{id}/start` |
| running | paused | `POST /sessions/{id}/pause` |
| paused | running | `POST /sessions/{id}/resume` |
| running / paused | finished | `POST /sessions/{id}/finish` 或到达 end_ts |
| running / paused | abandoned | 超时 24h 无活动 |

非法转换返回 `SESSION_STATUS_INVALID`。

### 4.3 创建流程（详细）

```
POST /api/v1/sessions
    ↓
EDGE 层鉴权 + 参数校验
    ↓
路由到某 SO 实例（基于一致性 Hash）
    ↓
SO 执行：
  1. 校验 scenario 存在
  2. 校验模型可用（调 infer-svc 的 /models/{id}/health）
  3. 创建 Session 对象（内存）
  4. 写入 sessions 表（Postgres）
  5. 存会话状态到 Redis（key: session:{id}:state）
  6. 建立 WebSocket room
  7. 返回 { session_id, ws_url }
```

### 4.4 结束流程（详细）

```
POST /api/v1/sessions/{id}/finish
    ↓
SO 执行：
  1. 状态切到 finished（不可逆）
  2. 取消所有 infer-svc 订阅
  3. 计算最终 portfolio（每个参与者）
  4. 生成 debrief
      ├── 收益 / 回撤 / 夏普 / 胜率
      ├── 交易日志
      ├── PK 模式：参与者对比 + 决策一致度
      ├── Copilot 模式：采纳/拒绝日志
      └── AI 洞察（调 LLM 生成）
  5. 写 sessions.summary
  6. 若 data_collection_opt_in：生成 training_samples（label 异步回填）
  7. 揭晓 scenario_reveal（盲测）
  8. 广播 "session_finished"
  9. 关闭 WebSocket room
  10. 清理 Redis 状态
```

## 5. 时钟模型

### 5.1 模拟钟 vs 墙钟

```
墙钟（Wall）          模拟钟（Sim）
─────────────         ─────────────
10:00:00 ──→ tick      2020-02-05 09:30
10:00:02 ──→           2020-02-05 09:35
10:05:00 ──→ tick      2020-02-05 09:40
10:05:15 ──→ 下单成功
10:15:00 ──→ tick      2020-02-05 09:45
...
```

**核心区别**：模拟钟推进**不是等时**的。用户思考 5 秒 / 5 分钟 / 1 小时都可以，模拟钟都在原地不动。

### 5.2 推进规则（伪代码）

```typescript
function advanceClock(session: Session, steps: number = 1): Clock {
  const c = session.clock;
  let newTs = addDuration(c.current_ts, c.granularity, steps);

  // 约束 1：不能超过 end_ts
  if (newTs > c.end_ts) newTs = c.end_ts;

  // 约束 2：严格单调
  if (newTs <= c.current_ts) {
    throw new Error('Clock must be strictly increasing');
  }

  // 约束 3：跳过非交易时段（分钟级）
  if (c.granularity === '5m' || c.granularity === '1h') {
    newTs = skipNonTradingPeriods(newTs);
  }

  c.current_ts = newTs;
  c.current_tick++;
  return c;
}
```

### 5.3 非交易时段

分钟级粒度需跳过：
- 非交易日（周末、法定节假日）
- 交易日的非交易时段（11:30~13:00 中午休市）
- 收盘后 15:00 → 跳到下一交易日 09:30

由 `trading_calendar` 模块统一维护，从 `market_snapshots` 或专门的交易日历表读取。

### 5.4 回放速度

| 用户操作 | 墙钟 | 模拟钟 |
|---------|------|-------|
| 暂停 | 继续流逝 | 冻结 |
| 手动推进 | 用户触发时推进 | +1 step |
| 自动回放（1x/2x/5x） | 前端定时器触发 | 每次 +1 step |
| 快进 | 一次请求 | 一次 +N steps |

无论哪种方式，**模拟钟语义始终是"当前数据截止时间"**。

## 6. 数据广播机制

### 6.1 差量广播

每次 tick 推一个差量包：

```typescript
interface TickPayload {
  clock: { current_ts: string; current_tick: number; };
  new_klines: KLine[];           // 此 tick 新增的 K 线
  new_news: NewsItem[];           // 此时间窗发布的新新闻
  portfolio_updates: Record<ParticipantId, { total_value: number; total_return: number; }>;
  signals?: Signal[];             // 仅 copilot_observer / copilot_interactive 模式
}
```

**为什么不用全量**：全量广播数据量会爆炸（后期几十 KB/tick）。差量平均 < 1KB/tick，带宽友好。前端已是状态化 React 应用，维护差量是本职。

### 6.2 消息顺序保证

每条消息带 `sequence`（会话内严格递增）：

```json
{ "type": "tick", "sequence": 42, "payload": {...} }
```

前端检测到 sequence 不连续，请求补发：

```json
{ "type": "request_replay", "from": 42, "to": 45 }
```

SO 从 Redis 的事件缓冲区（`session:{id}:events:queue`）取补发。

### 6.3 订阅模型

```typescript
interface Subscription {
  connection_id: string;
  session_id: string;
  participant_id: string | null;   // null 表示匿名观察
  filters?: {
    event_types?: string[];        // 只订阅某些事件
    participant_ids?: string[];    // 只看某些参与者
  };
}
```

一个会话可能有多个订阅（同一用户的多设备同时打开）。SO 维护 per-session 的订阅列表。

### 6.4 广播通道降级

```
WebSocket（首选）
    │ 连不上
    ↓
SSE（Server-Sent Events）
    │ 连不上
    ↓
HTTP 轮询（GET /sessions/{id}/events?since=N）
```

三种通道协议一致（消息格式相同），只是传输层不同。

## 7. 参与者模型

### 7.1 Human 参与者

```typescript
interface HumanParticipant {
  type: 'human';
  id: string;                    // user_id
  display_name: string;
  decision_method: 'ui';
}
```

下单路径：
```
前端按钮点击 → POST /sessions/{id}/orders → SO Match Engine → session_events
```

### 7.2 Model 参与者

```typescript
interface ModelParticipant {
  type: 'model';
  id: string;                    // model_id
  model_version: string;
  decision_timeout_seconds: number;  // 默认 5
  subscription_id: string;       // infer-svc 订阅
}
```

下单路径：
```
SO 广播 tick → SO 向 infer-svc 要 Signal → Signal 含 action+qty → SO 内部代模型下单 → Match Engine
```

**重要**：模型**自己不能直接下单**。必须由 SO 根据模型的 Signal 代为下单。这样才能保证：
- 所有订单都流经 Match Engine
- 同一 tick 内下单的时机对齐
- 防止模型绕过 as_of 拿到未来数据

### 7.3 Observer 参与者

只订阅信号，不下单。用于：
- 研究员观察模型在历史情境下的表现
- 模型 A/B 测试

### 7.4 Participant 的 Portfolio

每个参与者独立持有 Portfolio：

```typescript
interface Portfolio {
  cash: number;
  positions: Position[];
  total_value: number;          // cash + Σ(position.market_value)
  total_return: number;         // (total_value - initial_capital) / initial_capital
  max_drawdown: number;         // 历史最大回撤
  peak_value: number;           // 用于回撤计算
}
```

SO 在每个 tick 后更新所有参与者的 Portfolio。

## 8. 防未来函数实现

### 8.1 红线

**任何参与者在 `current_ts = T` 时，都绝对不能看到 `dt > T` 的数据。**

违反这条红线 = 实验失效 = 平台失信。

### 8.2 三层防护

**第 1 层 · API 约定**
所有数据查询必须传 `as_of`：

```typescript
// 正确
await dataClient.getKlines({ symbol, as_of: session.clock.current_ts });

// 错误：缺 as_of
await dataClient.getKlines({ symbol });  // 拒绝
```

**第 2 层 · Enforcer 拦截**

SO 在调用任何下游服务前，由 `AsOfEnforcer` 统一注入：

```typescript
class DataClientProxy {
  constructor(
    private underlying: DataClient,
    private clock: Clock,
  ) {}

  async getKlines(params: GetKlinesParams): Promise<KLine[]> {
    // 强制用 current_ts 覆盖任何传入的 as_of
    const safeParams = { ...params, as_of: this.clock.current_ts };
    return this.underlying.getKlines(safeParams);
  }
}
```

SO 传给 Infer Client / Data Client 的都是 Proxy，参与者无法绕过。

**第 3 层 · CI 级单元测试**

CI 必跑的测试：
```typescript
test('infer-svc must reject requests with as_of in the future', async () => {
  const session = createSession({ current_ts: '2020-02-05' });
  const attempt = () => inferClient.predict({
    model_id: 'xxx',
    symbols: ['600519'],
    as_of: '2020-02-10',  // 超前
  }, session);

  await expect(attempt).rejects.toThrow('INFERENCE_FUTURE_LEAK_DETECTED');
});

test('model cannot access data beyond session clock', async () => {
  // 装入一个"作弊模型"，故意请求未来数据
  const cheatModel = loadCheatModel();
  const result = await session.runTick();

  // 作弊应被拦截，session 状态标记为 aborted
  expect(result.status).toBe('aborted');
  expect(result.error).toContain('FUTURE_LEAK');
});
```

### 8.3 边界情况

**财报的 `announce_date`**
- 财报期 `2024-Q1` 的公告日期可能是 `2024-04-26`
- `as_of='2024-04-25'` 时**不能**看到 2024-Q1 财报
- `as_of='2024-04-26'` 时**可以**看到（严格小于？还是小于等于？详见下文）

**严格小于 vs 小于等于**
统一采用**严格小于**：`as_of=T` 表示能看到 `dt < T` 的数据。
好处：避免时钟精度引起的歧义（如 `15:00:00` 是否算收盘）。
对应 SQL：`WHERE dt < '2024-04-26T00:00:00Z'`

**新闻的 `published_at`**
精确到分钟甚至秒。`as_of='2024-01-23T10:00:00Z'` 只能看到 10:00 之前的新闻。

## 9. 撮合引擎

### 9.1 模式开关

```typescript
interface MatchEngineOptions {
  mode: 'simple' | 'realistic';
}
```

- **simple**：假设完美成交，按 K 线收盘价成交
- **realistic**：应用滑点、费率、T+1、涨跌停

MVP 用 simple，逐步切到 realistic。

### 9.2 订单流程（realistic）

```
收到订单 Order(symbol, side, qty, price, type)
    ↓
1. 校验
   ├── 股票是否在股票池
   ├── 股票是否停牌
   ├── T+1 检查（当日买入不能当日卖出）
   ├── 现金/持仓是否足够
   └── 涨跌停限制
    ↓
2. 滑点模拟
   ├── market 单：按当前买一/卖一价 + 滑点
   └── limit 单：按指定价，若当前价不到达则 pending
    ↓
3. 成交
   ├── 扣费（佣金 + 印花税）
   ├── 更新 portfolio
   └── 写 orders 表 + session_events
    ↓
4. 广播订单状态
```

### 9.3 默认费率

```typescript
const DEFAULT_RULES = {
  commission_rate: 0.0003,        // 万三
  stamp_tax: 0.001,               // 千一，仅卖出
  min_commission: 5,              // 单笔最低 5 元
  slippage: 0.0005,               // 万五
  t_plus_1: true,                 // A 股 T+1
  allow_limit_order: false,       // 涨跌停挂单
};
```

### 9.4 撮合次序

同一 tick 内，如果多个参与者同时下单：
- **FIFO**：按 `created_at` 墙钟先后
- 模型的 `created_at` = SO 收到 Signal 的时间
- 人的 `created_at` = SO 收到 HTTP 请求的时间

没有"模型先手"的说法。

## 10. 四种模式的差异实现

所有模式共用同一套 SO 框架，差异只体现在 **Session Manager 的调度策略**和**前端 UI**。

### 10.1 模式 ① Solo（纯用户）

- 无模型参与者
- tick 只推 data 更新，不推 signal
- 撮合只处理用户订单

### 10.2 模式 ② Copilot Observer（副驾）

- 有 1 个模型参与者，但 `role='observer'`（不下单）
- 每个 tick SO 向 infer-svc 请求 Signal
- Signal 推给前端副驾面板
- 模型不参与撮合

关键代码片段：
```typescript
function onTick(session: Session) {
  if (session.mode === 'copilot_observer') {
    const signals = await inferClient.predict({
      model_id: session.observers.model_id,
      symbols: session.scenario.symbols,
      as_of: session.clock.current_ts,
      include_explanation: true,
    });
    broadcast(session, { type: 'model_signal', signals });
  }
}
```

### 10.3 模式 ③ PK

- 用户 + 1 个模型，两者都独立下单
- 每个 tick SO 向 infer-svc 请求 Signal，并**立即代模型下单**
- 模型决策超时（默认 5s），超时视为"持有"
- 结束后生成对账和决策一致度

关键代码片段：
```typescript
async function onTick(session: Session) {
  if (session.mode === 'pk') {
    const signals = await inferClient.predict({...});

    for (const signal of signals) {
      if (signal.action !== 'HOLD') {
        const order = signalToOrder(signal, session);
        await matchEngine.place(order, { on_behalf_of: session.models[0] });
      }
    }
  }
}
```

**公平竞赛**：
- 同 tick 内人和模型独立下单
- 撮合按 FIFO（墙钟先后）
- 模型看不到人的订单，反之亦然
- 两者 portfolios 完全隔离

### 10.4 模式 ④ Copilot Interactive（协作）

- 用户 + 1 个模型（observer）
- 用户每次下单前调 `POST /sessions/{id}/copilot/suggestion`
- SO 请求模型信号 + 对比用户提议
- 前端弹窗展示"采纳/调整/拒绝"
- 记录决策日志到 `session_events`

数据闭环（模式 ⑤，隐式存在）：
- 每个 session 生成 `training_samples`
- 标签（label）需要等 N 日后的真实收益回填
- 回填任务由 `feature-svc` 的日批处理执行

## 11. 故障恢复

### 11.1 断线场景

**用户 WebSocket 断线**
- SO 维护 `connection_id` 和 `session_id` 的映射
- 60 秒内重连：恢复订阅，用 `sequence` 补发丢失事件
- 60 秒外重连：状态已从 Redis 失效，重新订阅

**SO 实例崩溃**
- Redis 保存会话状态（`session:{id}:state`）
- 新 SO 实例通过一致性 Hash 接管
- 从 Redis 加载状态 + 重建 WebSocket room
- 通知用户重连（`session_resumed`）

### 11.2 推理超时

```typescript
async function requestSignal(session: Session): Promise<Signal[]> {
  try {
    return await Promise.race([
      inferClient.predict({...}),
      timeout(session.models[0].decision_timeout_seconds * 1000),
    ]);
  } catch (e) {
    if (e instanceof TimeoutError) {
      recordEvent(session, {
        type: 'model_timeout',
        payload: { model_id: session.models[0].id },
      });
      return [];  // 空信号 = 持仓不变
    }
    throw e;
  }
}
```

### 11.3 撮合失败

- 现金不足 / 涨跌停 / 停牌等：订单状态 `rejected`
- 前端收到 `order_rejected` 事件，显示 Toast
- 会话继续，不中断

### 11.4 数据层不可用

- Postgres 短暂不可用：SO 继续运行，事件先入 Redis 队列
- Redis 不可用：降级到纯内存 + 持久写 PG（延迟增大但不丢数据）
- 两者都不可用：当前会话强制暂停，30 秒后重试，超 3 次则 abandon

## 12. 性能与扩展

### 12.1 单实例容量

在一台 2c4g 的节点上：
- 并发会话：~500
- WebSocket 连接：~1000（考虑多端）
- tick 吞吐：~500/秒
- 推理延迟目标：< 200ms（包括网络）

### 12.2 水平扩展

**粘性会话（Sticky Session）**：同一 session_id 的请求路由到同一实例。
- 用一致性 Hash：`hash(session_id) % N`
- 实例变化时少量 session 会被转移

**状态迁移**：实例缩容前，把 session 状态同步到 Redis + 触发重路由。

### 12.3 瓶颈分析

| 瓶颈 | 优化 |
|------|------|
| 推理延迟 | infer-svc 模型热缓存 + 预计算 |
| 数据库写 | 事件异步批量写 |
| WebSocket 连接数 | 多实例 + LB |
| Redis 内存 | 会话结束立即清理 |

### 12.4 监控指标

| 指标 | 告警阈值 |
|------|---------|
| `session_active_count` | - |
| `tick_latency_p99` | > 500ms |
| `infer_timeout_rate` | > 5% |
| `match_error_rate` | > 1% |
| `ws_reconnect_rate` | > 10% |
