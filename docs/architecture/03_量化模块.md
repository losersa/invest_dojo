## 1. 架构分层

```
┌─────────────────────────────────────────────────────────────┐
│                    UI 层（Next.js）                          │
│  因子库 / 训练 / 回测 / 模型市场 / 联动控制                     │
└─────────────────────────────────────────────────────────────┘
                              ↓ ↑
┌─────────────────────────────────────────────────────────────┐
│          Session Orchestrator（Node 长连接）                 │
│  管理人+模型共享同一份行情流 / 同步时钟 / 记录对账              │
└─────────────────────────────────────────────────────────────┘
                              ↓ ↑
┌──────────────────────────────────┬──────────────────────────┐
│    Next.js Route Handlers         │  Python 微服务 (FastAPI) │
│  · 用户/模型市场/权限              │  · feature-svc            │
│  · 轻业务逻辑                      │  · train-svc              │
│                                  │  · infer-svc              │
│                                  │  · backtest-svc           │
│                                  │  · monitor-svc            │
└──────────────────────────────────┴──────────────────────────┘
                              ↓ ↑
┌─────────────────────────────────────────────────────────────┐
│                   数据层                                     │
│  Supabase (PostgreSQL)     Redis Cache       S3/MinIO       │
└─────────────────────────────────────────────────────────────┘
```

### 分工原则
- **Node 做编排**：会话层、前端 API、轻业务逻辑
- **Python 做计算**：训练、回测、推理、特征计算
- **Supabase 管元数据**：用户、模型信息、会话、订单
- **Redis 管热数据**：因子缓存、会话状态、推理信号流
- **S3 管大文件**：模型文件、Notebook 快照、回测报告

## 2. 核心抽象

### 2.1 `Model` 接口

所有模型（无论来源）必须实现：

```python
class Model(ABC):
    name: str
    version: str
    input_features: List[str]   # 需要哪些因子
    output_type: Literal['classification', 'regression', 'rank']

    def predict(self, features: DataFrame) -> DataFrame: ...
    def to_signal(self, prediction: DataFrame) -> List[Signal]: ...
```

### 2.2 `Signal` 格式（统一信号）

```python
@dataclass
class Signal:
    timestamp: datetime       # 信号时间
    symbol: str               # 股票代码
    action: Literal['BUY', 'SELL', 'HOLD']
    confidence: float         # 0~1
    features: Dict[str, float]  # 支撑信号的关键特征值
    metadata: Dict            # 模型版本、推理耗时等
```

**重要**：联动层只认 Signal，不关心它来自哪里。

### 2.3 `Session` 接口

```python
@dataclass
class Session:
    id: str
    scenario_id: str
    user_id: str
    mode: Literal['solo', 'copilot', 'pk', 'observe']
    participants: List[Participant]
    clock: Clock
    initial_capital: float
    status: Literal['running', 'paused', 'finished']
```

```
└────────┬────────┘  └────────┬────────┘  └────────┬─────────┘  └────────┬─────────┘
         │                    │                    │                     │

                                        ↓
                               统一 Model 注册表
                                        ↓
                                  Signal 输出
```

### 3.2 ⭐ 平台官方模型扩展点（核心设计）

**关键原则**：官方模型和用户模型走**完全一样的注册路径**，只是 `owner='platform'`、多几个额外属性。这样：

1. **技术上零特殊路径**：调用、推理、监控都复用用户模型的代码
2. **完全透明**：用户能看官方模型的完整代码、因子、训练过程
3. **持续维护**：官方模型的再训练只是一次普通训练任务
4. **扩展性强**：后续要做付费精品只需加 `tier` 字段

#### 官方模型元数据扩展字段

```typescript
interface PlatformModelExtras {
  is_official: boolean;              // true
  official_tier: 'baseline' | 'experimental' | 'premium';
  official_category:                 // 官方模型流派分类
    | 'momentum_follower'
    | 'value_investor'
    | 'sentiment_trader'
    | 'multi_factor'
    | 'deep_learning';
  doc_url: string;                   // 详细文档地址
  source_code_url: string;           // 开源训练代码地址
  retrain_schedule: 'quarterly' | 'monthly' | 'manual';
  last_retrained_at: Date;
  maintainer: string;                // 维护者
  deprecation_notice?: string;       // 弃用提示
  methodology: {                     // 方法论描述
    thesis: string;                  // 核心假设
    features_used: string[];
    target: string;
    training_window: [Date, Date];
    known_limitations: string[];
  };
}
```

#### 官方模型发布流程

```
1. 算法团队在内部环境训练（用平台自己的训练服务）
       ↓
2. 通过标准化评估（验证集 AUC > 基线 / 年化收益 > 上证+5%）
       ↓
3. 生成官方模型元数据（tier、category、doc、methodology）
       ↓
4. 写到 models 表，owner='platform', is_official=true
       ↓
5. 自动暴露到"模型市场 > 官方模型"板块
       ↓
6. 季度定时任务：用最新数据重训，版本号递增
```

#### 官方模型的数据库视图

```sql
-- 只查官方模型的视图
CREATE VIEW official_models AS
SELECT
    m.*,
    m.metadata->>'official_tier' AS tier,
    m.metadata->>'official_category' AS category,
    m.metadata->>'maintainer' AS maintainer
FROM models m
WHERE m.owner = 'platform' AND m.visibility = 'public';

-- 用户模型 + 官方模型的统一排行榜（按调用量）
CREATE VIEW model_leaderboard AS
SELECT
    id, name, owner,
    CASE WHEN owner = 'platform' THEN 'official' ELSE 'user' END AS source,
    usage_count,
    metadata->>'last_30d_return' AS recent_return
FROM models
WHERE visibility = 'public'
ORDER BY usage_count DESC;
```

#### 未来扩展预留

虽然 MVP 不做，但字段已为这些预留：
- **付费精品模型**：`official_tier = 'premium'` + `price_per_call`
- **限时实验**：`experimental` tier 可随时下线
- **社区共建**：引入 `maintainer_type = 'community'` 但走官方评审流程
- **A/B 测试**：同一个 category 的多个官方模型并行发布，观察用户偏好

### 3.3 用户上传模型的隔离机制

（本质上官方模型也走同一个推理沙箱，只是权限上可以信任）

```
Upload Flow:
    .pkl 文件
        ↓
    Smoke Test（沙箱跑 3 条样本）
        ↓
    通过 → 注册到 models 表
    失败 → 返回具体错误
```

Infer Flow:
```
Request → Infer Worker → 沙箱容器（gVisor）
                         ↓
                    加载模型（受限权限）
                    只能访问平台数据 API
                    禁止网络/文件系统
                         ↓
                    返回 Signal
```

## 4. 数据层与特征工程（三层架构）

### 4.1 数据层扩展（基于现有 Supabase）

**现有表**：`klines_all`, `news`, `scenarios`

**新增表**：
- `fundamentals` — 财务数据（季度更新）
- `market_snapshot` — 市场快照（指数/板块/北向资金）
- `feature_values` — 预计算的因子值矩阵（稀疏）

详见 [01_数据层.md](./01_数据层.md)。

### 4.2 三层特征体系

#### Layer 1 · 因子库（200+ 内置）

```yaml
# 因子定义格式（YAML）
id: ma_cross_20_60
name: 20日均线上穿60日均线
category: technical
formula: |
  MA(close, 20) > MA(close, 60) and
  MA(close, 20).shift(1) <= MA(close, 60).shift(1)
output_type: boolean
lookback_days: 60
version: 1
maintainer: platform
```

#### Layer 2 · 可视组合器（P1）
- 前端拖拽 → 生成 AST
- 后端把 AST 转为 Pandas 表达式
- 支持所有 Layer 1 因子作为基础块

#### Layer 3 · JupyterLite Notebook（P2）
- 浏览器内运行 Python（Pyodide）
- 提供平台 SDK（`investdojo` 包）
- 用户代码不上服务器，只调用数据/训练 API

### 4.3 计算策略

| 因子类型 | 计算时机 | 存储位置 |
|---------|---------|---------|
| 热门内置因子（200个） | 夜间预计算 | `feature_values` 表 |
| 用户自定义因子 | 首次使用时计算 + 缓存 2h | Redis |
| 实时信号（副驾模式） | 每次 tick 即时算 | 不存 |

## 6. 回测引擎（backtest-svc）

### 6.1 两种模式

| 模式 | 说明 | 延迟 | 使用场景 |
|------|------|------|---------|
| **Fast** | 向量化，假设完美成交 | < 30s | 调参、批量对比 |
| **Realistic** | 事件驱动撮合，有滑点/费率/涨跌停 | 秒~分钟 | 最终评估、联动前准备 |

### 6.2 关键输出

- 净值曲线（带基准对比）
- 收益/回撤/夏普/卡玛/IC
- 分段表现（牛市/熊市/震荡市）
- 特征重要性 SHAP
- 完整交易日志

## 8. 监控服务（monitor-svc）

### 8.1 监控维度

| 维度 | 告警条件 | 动作 |
|------|---------|------|
| 推理表现 | 过去 30d 收益 vs 上证 < -10% | 标红 + 站内信 |
| 数据漂移 | 特征 KL 散度 > 阈值 | 建议重训 |
| 特征可用性 | 某因子连续 3d 计算失败 | 触发补数 |
| 推理延迟 | P99 > 500ms | 扩容 |
| 调用量 | 7d 无调用 | 移出热缓存 |

### 8.2 官方模型的特殊监控

官方模型作为平台的"脸面"，有更严格的监控：
- 每日自动回测并对比历史表现
- 收益率连续 2 周 < 上证，触发算法团队人工 review
- 重大偏离（如回撤超预期 2 倍）自动暂停并通知维护者

## 9. 会话编排层（联动核心）

详见 [04_联动机制.md](./04_联动机制.md)。

### 9.1 核心职责

1. **时钟同步**：人和模型看到的"当前时间"严格一致
2. **数据广播**：每次 tick，把新 K 线+新闻推给所有参与者
3. **事件记录**：人下单、模型信号、成交结果统一入库
4. **结束对账**：生成复盘报告

### 9.2 严格防未来函数

- 联动模式下，所有数据查询都强制传当前会话时钟
- 推理服务在沙箱中只能访问 `as_of <= session.clock.current_ts` 的数据
- CI 级单元测试保证红线不破

## 10. 数据库增量设计

完整 DDL 见 [01_数据层.md](./01_数据层.md)。本章仅列新增表：

### 10.1 新增表清单

- `factor_definitions` — 因子定义
- `feature_values` — 预计算因子值
- `models` — 模型注册表（支持平台官方模型）
- `backtests` — 回测结果
- `sessions` — 会话
- `session_events` — 会话事件日志
- `training_samples` — 用户训练样本（Mode ④）

### 10.2 `models` 表关键字段

```sql
CREATE TABLE models (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    version TEXT NOT NULL,
    owner TEXT NOT NULL,                 -- user_xxx / 'platform'
    source TEXT NOT NULL,                -- internal/uploaded/external/official
    input_features JSONB,
    target TEXT,
    training_range JSONB,
    validation_metrics JSONB,
    file_path TEXT,
    visibility TEXT DEFAULT 'private',   -- private/public
    metadata JSONB,                      -- ⭐ 存 PlatformModelExtras 等扩展字段
    created_at TIMESTAMPTZ DEFAULT NOW(),
    last_used_at TIMESTAMPTZ,
    usage_count INT DEFAULT 0
);

-- 官方模型快速查询索引
CREATE INDEX idx_models_official ON models(owner) WHERE owner = 'platform';
CREATE INDEX idx_models_visibility ON models(visibility, usage_count DESC);
```

## 13. 风险与权衡

| 风险 | 影响 | 缓解 |
|------|------|------|
| 推理延迟大 | 副驾体验差 | 预计算 + Redis 缓存 |
| 用户模型安全 | 系统被攻击 | gVisor 沙箱 + 配额 |
| 训练阻塞 | 用户等太久 | 队列 + 进度通知 |
| 官方模型失效 | 平台信誉受损 | 严格监控 + 自动告警 |
| Supabase 存储成本 | 全 A 股数据量大 | 冷数据迁 S3 |
