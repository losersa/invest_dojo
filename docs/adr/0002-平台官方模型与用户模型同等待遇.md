# ADR 0002 · 平台官方模型与用户模型同等待遇

> 日期：2026-04-28
> 状态：Accepted
> 决策者：算法 + 产品 + 工程
> 上下文：量化模块需要提供官方基线模型供新用户参考

---

## 背景

量化模块规划中，平台需要提供若干官方模型（"动量追踪者"、"价值投资者"等），作用：
1. 新用户的默认副驾
2. 展示平台技术能力
3. 作为用户模型的性能基准

关键问题：**官方模型在技术实现上应不应该特殊对待？**

## 选项对比

### 选项 A · 独立子系统
- 单独的"官方模型"表、独立的推理路径、独立的管理后台
- ✅ 可以做精细化定制（如更高 SLA）
- ❌ 维护两套代码
- ❌ 官方和用户模型无法直接对比
- ❌ 将来如果做社区精选模型需要第三套

### 选项 B（选中）· 统一路径
- 官方模型和用户模型共用 `models` 表、共用推理服务
- 官方模型只是 `owner = 'platform'`、多几个元数据字段
- ✅ 零特殊路径，维护成本低
- ✅ 完全透明（用户能看官方模型的代码/特征）
- ✅ 统一排行榜（人机混排体现"道场"概念）
- ✅ 扩展性强：未来做付费精品只需加 `tier` 字段
- ❌ 需要靠权限控制避免普通用户冒充官方

### 选项 C · 仅元数据标记
- 只在前端展示时给官方模型加"官方"标签，后端完全无差
- ✅ 更简单
- ❌ 不利于未来的官方模型特殊策略（如强制监控、季度重训）

## 决策

采用**选项 B · 统一路径 + 差异化元数据**。

具体实现：
- `models.owner = 'platform'` 区分
- `models.metadata` JSONB 存 `PlatformModelExtras` 扩展字段
- 视图 `official_models` 提供快捷查询
- 推理/回测/监控流程**完全复用**用户模型路径
- 权限：只有 `role = 'platform_admin'` 的 DB 账号能创建 `owner = 'platform'` 的记录

### 扩展字段契约

```typescript
interface PlatformModelExtras {
  is_official: true;
  official_tier: 'baseline' | 'experimental' | 'premium';
  official_category: 'momentum_follower' | 'value_investor' | ...;
  doc_url: string;
  source_code_url: string;         // ⭐ 开源的核心
  retrain_schedule: 'quarterly' | 'monthly' | 'manual';
  last_retrained_at: Date;
  maintainer: string;
  methodology: {
    thesis: string;
    features_used: string[];
    target: string;
    training_window: [Date, Date];
    known_limitations: string[];
  };
}
```

## 特殊处理（不影响主路径）

虽然主流程完全统一，少数地方有差异：

1. **前端展示**：模型市场有独立的"⭐ 官方模型"板块置顶
2. **监控**：官方模型触发更严格的告警阈值
3. **生命周期**：官方模型不可删除、季度自动重训
4. **默认推荐**：新用户首次进入副驾模式默认加载平台推荐的官方模型

这些差异**都在业务层实现**，不影响数据层/推理层的统一性。

## 后果

**正面**：
- 开发成本低（复用现有路径）
- 用户心智统一（所有模型一视同仁对比）
- 透明可学习（官方模型的训练代码也开源）
- 扩展性强（付费、社区精选、A/B 实验都有预留字段）

**负面**：
- 需要管控 `owner='platform'` 的创建权限
- 初期官方模型更新依赖算法团队手动操作

**风险缓解**：
- DB RLS 策略：只有 service_role 能写 `owner='platform'`
- CI 中加入官方模型发布检查脚本（校验元数据完整性）
- 运行时监控：官方模型若表现异常立刻站内信通知全体用户

## 衍生决策

- 将来的社区精选模型走类似路径：`owner='community_curator_123'`，带 `curated_by_platform: true` 标签
- 将来的付费模型走 `tier: 'premium'` + `price_per_call` 字段，推理时做额度核销
