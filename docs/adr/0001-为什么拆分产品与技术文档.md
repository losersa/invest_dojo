# ADR 0001 · 为什么拆分产品与技术文档

> 日期：2026-04-28
> 状态：Accepted
> 决策者：产品 + 工程
> 上下文：项目初期文档混合了产品需求与技术方案，预感未来变更会频繁

---

## 背景

InvestDojo 处于 0 到 1 的早期阶段，文档演进特点是：
1. **产品需求变化快**：模块优先级、用户故事、功能边界每周都在调整
2. **技术架构变化慢**：一旦定下基础设施不会频繁改
3. **API 变化中等**：新增比修改多，但要保证前后端契约稳定

此前把产品描述（"用户画像/用户故事"）和技术实现（"数据库设计/服务拓扑"）放在同一份文档里，导致：
- 改需求要动整份文档，怕改错技术部分
- 工程师看技术时被产品描述干扰
- 产品同事看需求时被大量代码吓到
- 文档版本管理混乱（一次改动可能是产品调整也可能是架构演进）

## 决策

采用**四层文档结构**：

```
docs/
├── product/         产品需求层（PM/UX 高频维护）
├── architecture/    技术架构层（工程师低频维护）
├── api/             接口契约层（前后端共同，中频维护）
└── adr/             架构决策记录（一次性写入）
```

明确边界：
- PRD 不写技术方案
- 架构文档不写需求
- API 独立版本化
- ADR 记录为什么

## 选项对比

### 选项 A（选中）· 分层独立
- ✅ 各角色职责清晰
- ✅ 变更影响范围小
- ✅ 符合行业最佳实践（Stripe/Twilio 都这么做）
- ❌ 初期写作成本略高（要在多处维护交叉引用）

### 选项 B · 单一大文档
- ✅ 写起来方便
- ❌ 改起来混乱
- ❌ 不利于团队协作

### 选项 C · 按模块划分（每个模块一份混合文档）
- ✅ 一个功能相关内容都在一起
- ❌ 跨模块查找痛苦
- ❌ 无法区分"这是产品共识"还是"这是技术选择"

## 后果

**正面**：
- 产品文档可由产品/PM 独立维护，工程师无需 review
- 架构文档稳定，可作为新人入职必读
- API 文档可自动生成/校验（FastAPI OpenAPI 直接对齐）
- ADR 成为团队"机构记忆"

**负面**：
- 首次拆分成本：~2h 工作量
- 需要维护跨文档引用
- 新人上手前需要看 README 了解结构

**风险缓解**：
- 每份文档头部必须有"对应 PRD / 对应 API"的反向链接
- CI 检查死链（TODO）
- 每季度 review 文档结构是否还合适
