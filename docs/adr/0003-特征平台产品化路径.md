# ADR 0003 特征平台产品化路径

- **状态**：Accepted
- **日期**：2026-05-01
- **作者**：@losersa + AI
- **关联**：[特征平台架构图](../../.workbuddy/brain/.../feature-platform-architecture.html) · [产品 PRD §3.1 因子库](../product/02_量化模块_PRD.md#31-因子库p0)

---

## 背景

InvestDojo 当前（2026-05-01）已完成 Epic 3 前 5 个任务：

- T-3.01 DSL 解析器（15 内置函数，Pratt 法）
- T-3.02 因子计算引擎（pandas 向量化，3000 股 × 252 天 < 3s）
- T-3.03 80 个技术因子
- T-3.04 120 个基本面/估值/情绪因子
- T-3.05 定时批量计算 + 全量回填脚本

技术上「因子库」已经可用，但**只能工程师通过 YAML + Git commit 发布因子**，用户/研究员自己不能。

产品期望（2026-05-01 讨论）：

> 用户想把 factors 模块升级成真正的"特征平台"：在 UI 里写公式 → 选调度 → 自动计算 + 上线 + 订阅，不需要工程师手动写代码。

这是一个**明确的长期产品方向**，需要决定：

1. 要不要把 factors 模块产品化成"特征平台"？
2. 如果要，短/中/长期路线怎么切？避免过度设计的同时保证未来能扩展？

---

## 决策

### 1. 确认方向：factors 模块产品化为"InvestDojo 特征平台"

**Yes**，理由：

- 因子/特征是**量化投资的核心资产**，能自助生产意味着平台 UGC 飞轮能跑起来
- 技术基础已经到位（DSL + 引擎 + 200 个示范 + 调度），离产品化只差 UI 和若干服务层能力
- 与主业务（模拟炒股、回测）天然耦合：用户造的因子可以直接被策略使用
- 未来有 B 端变现潜力（给券商/私募卖 SaaS，参考 Tecton / 字节 ByteHouse）

### 2. 五层架构模型

参考业界标杆（Tecton / Feast），InvestDojo 特征平台定义为五层栈：

| 层 | 职责 | MVP 现状 | v1 目标 |
|---|---|---|---|
| **① Definition** | 用户怎么描述因子 | DSL 解析器 ✅ | + Monaco 编辑器 UI |
| **② Scheduling** | 什么时候算 | Celery Beat 全局 17:00 ✅ | + 每因子独立 cron |
| **③ Computation** | 怎么算 | pandas 批量向量化 ✅ | 保持 |
| **④ Storage** | 存哪 | PostgreSQL + 分区 ✅ | + Redis 在线缓存 |
| **⑤ Serving** | 谁消费 | 3 个 REST API ✅ | + 前端浏览/编辑 UI |

### 3. 版本路线

**v1（2-3 周）**：产品级平台 — 用户能写、能定时、能分享

- T-3.06 因子库 API 完整（CRUD、compare、history）
- T-3.07 因子库前端页（列表 + 详情 + 排行）
- T-3.08 因子编辑器 UI（Monaco + 语法高亮 + 实时预览）
- T-3.09 每因子独立 cron 调度（Beat 动态加载）
- T-3.10 Redis 在线缓存（查因子 < 10ms）
- T-3.11 因子版本化（改公式不覆盖历史）
- T-3.12 权限分享（public / private / shared + user_id）

**v2（1-2 月）**：企业级 — 团队可用，强调度，多数据源

- 多数据源接入（新闻、龙虎榜、北向、社交）
- 实时流式计算（Kafka/NATS 订阅 Tick）
- Point-in-Time 读取（严格时间对齐，回测防未来函数）
- Feature Pipeline DAG（因子间依赖 + topo 排序）
- 自动 backfill 调度器（新因子上线 → 后台补历史）
- 数据质量规则（NaN 率、分布漂移、分位异常）
- A/B 分桶 + 订阅通知

**v3（3-6 月）**：生态型 — 对外输出，SaaS 化

- Marketplace（发布 / 订阅 / 打赏）
- 多租户 SaaS（给券商/私募）
- 因子 IDE（项目化 + Git 集成 + CI）
- 自动性能优化（合并同构因子缓存）
- 联邦学习（因子值不出库，只返聚合结果）

## 备选方案（已否决）

### A. 维持工程师维护 YAML 模式

- ✅ 简单、稳定
- ❌ UGC 飞轮起不来
- ❌ 用户想试新因子必须找人改代码

### B. 直接一步到位做 v3 Marketplace

- ✅ 愿景宏大
- ❌ **严重违反短期原则**，v1 都没有、没有用户群、没有验证需求
- ❌ 技术栈（多租户、计费、联邦学习）投入产出比太低

### C. 接入 Feast / Tecton 开源引擎

- ✅ 有成熟生态
- ❌ **Feast 不支持我们的 DSL 风格**，它的定义语言是 Python class，我们 DSL 用户更友好
- ❌ Tecton 商用、不开源，Feast 偏 ML 特征而不是金融因子
- ❌ 我们的场景特殊（PIT、财报 announce_date 前向填充、A 股涨跌停规则），外部工具不够用

## 后果

### 正面

- 明确的长期产品方向，后续开发不盲目
- 决定把资源聚焦在 v1，短期目标清晰
- 五层架构模型给未来每个 task 都有"我在哪一层"的定位
- v2/v3 作为长期愿景，但**现在不投入**，避免过度设计

### 负面 / 风险

- UI 投入会比纯后端任务大（Monaco、图表、表单、权限）
- Redis、Beat 动态加载、版本化都增加运维复杂度
- 前期用户量少时，Marketplace 可能冷启动失败（v3 再处理）

### 约束

- **短期原则**：先把 T-3.06 / T-3.07 / T-3.08 做出来能 demo，再规划 T-3.09+
- 每次讨论"新能力"时，先问"这是 v1/v2/v3？"，不是 v1 的别占 v1 预算
- 外部依赖少加（Redis 是唯一新引入的，Kafka/Ray 留给 v2）
