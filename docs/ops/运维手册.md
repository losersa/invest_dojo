## 1. 基础信息

### 1.1 机器角色

| 机器 | 角色 | 位置 |
|---|---|---|
| **Windows** (hostname: `investdojo`, IP: `192.168.1.5`) | 数据库宿主 + 可选开发 | 固定在家里 |
| **Mac** | 主开发机 | 便携 |

两机必须在**同一 Wi-Fi**（目前 `CMCC-e7sa-5G`，**Private** profile）才能互通。

### 1.2 核心目录

**Windows**：
```
E:\project\ownproject\invest_dojo\                ← git 仓库
  └─ investdojo\infra\supabase-lite\              ← 栈定义
     ├─ docker-compose.yml
     ├─ .env                                       ← 密钥（不进 git）
     └─ data\db\                                   ← Postgres 数据目录（bind mount）

E:\Docker\DockerDesktopWSL\                       ← Docker 引擎数据（C 盘已空）
```

**Mac**：
```
~/你的仓库路径/invest_dojo/                       ← 对应的 git clone
```

### 1.3 技术栈 · 一页图

```
Mac 端                          Windows 端
─────────                       ──────────
Next.js web  :3000              Docker Desktop
  │ @supabase/ssr                 ├─ investdojo-db (PG 15.8)     :5432
Node orch    :8787                ├─ investdojo-rest (PostgREST) internal
  │ @supabase/supabase-js         ├─ investdojo-auth (GoTrue)    internal
Python svc   :8000-8005           └─ investdojo-kong             :8000
  │ httpx → PostgREST
  │
  └─ 所有请求 → http://investdojo.local:8000
```

## 2. 日常操作

### 2.1 Windows 端：启停栈

```powershell
cd E:\project\ownproject\invest_dojo\investdojo\infra\supabase-lite

# 启动（首次会自动生成密钥；之后重启无副作用）
.\scripts\up.ps1

# 停止（数据保留）
.\scripts\down.ps1

# 查状态
docker compose ps

# 查某服务日志
docker compose logs -f db
docker compose logs -f rest
docker compose logs -f auth
docker compose logs -f kong

# 重启单服务
docker compose restart auth
```

**自动拉起**：容器都配了 `restart: unless-stopped`，Windows 重启后 Docker Desktop 启动会自动把它们拉起来。

### 2.2 Windows 端：进 Postgres

```powershell
cd E:\project\ownproject\invest_dojo\investdojo\infra\supabase-lite
$pgPwd = (Select-String -Path .env -Pattern '^POSTGRES_PASSWORD=').Line.Substring(18)
docker compose exec -T -e PGPASSWORD=$pgPwd db psql -U postgres -d postgres

# 或者快速查一条
docker compose exec -T -e PGPASSWORD=$pgPwd db psql -U postgres -d postgres -c "SELECT count(*) FROM public.symbols"
```

### 2.3 Mac 端：连 Windows 的 PG

```bash
# 基础测试
ping investdojo.local
curl -s http://investdojo.local:8000/auth/v1/health

# 命令行 psql（需要装 libpq）
PGPASSWORD='<POSTGRES_PASSWORD>' psql -h investdojo.local -U postgres -d postgres

# DBeaver / pgAdmin 图形工具同样用 investdojo.local:5432
```

### 2.4 Mac 端：日常开发

```bash
cd investdojo

# 启动 Python 服务（data/feature/train/infer/backtest/monitor）
cd python-services && make dev

# 启动前端
cd apps/web && pnpm dev

# 跑测试
cd python-services && make test-unit
make test-integration
bash scripts/smoke.sh
```

### 3.3 数据迁移（Cloud → 本地）

**场景**：首次部署后，把 Supabase Cloud 上的历史数据搬到本地。

**在 Mac 端执行**（Windows agent 配合）：

```bash
# 1) 从 Supabase Cloud 导出（只导数据，schema 本地已建好）
pg_dump "postgresql://postgres:<CLOUD_PWD>@db.adqznqsciqtepzimcvsg.supabase.co:5432/postgres" \
    --data-only \
    --schema=public \
    --disable-triggers \
    -Fc -f ~/investdojo-cloud.dump

# 预计：500MB-1GB，百兆网 30-40 分钟

# 2) 恢复到 Windows 本地（走 PG 协议直连）
PGPASSWORD='<WIN_PG_PWD>' pg_restore \
    -h investdojo.local -p 5432 \
    -U postgres -d postgres \
    --data-only --disable-triggers --jobs=4 \
    ~/investdojo-cloud.dump

# 3) 校验行数
PGPASSWORD='<WIN_PG_PWD>' psql -h investdojo.local -U postgres -d postgres <<'SQL'
SELECT 'symbols', count(*) FROM symbols
UNION ALL SELECT 'klines_all', count(*) FROM klines_all
UNION ALL SELECT 'news', count(*) FROM news
UNION ALL SELECT 'market_snapshots', count(*) FROM market_snapshots
UNION ALL SELECT 'scenarios', count(*) FROM scenarios;
SQL
```

**基准值**（来自 `docs/product/99_MVP_Sprint0.md`）：
- symbols: 5524
- industries: 102
- klines_all: ~5.7M（日K 5.6M + 5m 108K + 老 73K）
- news: 49
- market_snapshots: 2995
- scenarios: 4
- factor_definitions: 5

### 3.4 恢复整库（如果数据损坏）

```powershell
# 1) 停栈
docker compose down

# 2) 清空数据目录
Remove-Item -Recurse -Force .\data\db\*

# 3) 重启（会重跑 init SQL）
.\scripts\up.ps1

# 4) 应用业务 migrations
# 先应用 legacy 表
docker cp init\01_legacy_tables.sql investdojo-db:/tmp/legacy.sql
$pgPwd = (Select-String -Path .env -Pattern '^POSTGRES_PASSWORD=').Line.Substring(18)
docker compose exec -T -e PGPASSWORD=$pgPwd db psql -U postgres -d postgres -f /tmp/legacy.sql

# 然后手动逐个跑 migrations/000-005
foreach ($m in Get-ChildItem ..\..\migrations\*.sql | Sort-Object Name) {
    docker cp $m.FullName "investdojo-db:/tmp/$($m.Name)"
    docker compose exec -T -e PGPASSWORD=$pgPwd db psql -U postgres -d postgres -v ON_ERROR_STOP=1 -f "/tmp/$($m.Name)"
}

# 5) 应用 post-gotrue 补丁
docker cp .\scripts\post-gotrue-patch.sql investdojo-db:/tmp/patch.sql
docker compose exec -T -e PGPASSWORD=$pgPwd db psql -U supabase_auth_admin -d postgres -f /tmp/patch.sql

# 6) 从最新 pg_dump 恢复数据
docker cp .\data\db-backup\<latest>.dump investdojo-db:/tmp/restore.dump
docker compose exec -T -e PGPASSWORD=$pgPwd db pg_restore -U postgres -d postgres --data-only --disable-triggers /tmp/restore.dump
```

## 4. 排障指南

### 4.1 Mac 连不上 Windows

```bash
# 诊断脚本
ping investdojo.local                                       # mDNS 解析
ping 192.168.1.5                                            # 直接 IP
nc -zv investdojo.local 8000                                # Kong 端口通不通
nc -zv investdojo.local 5432                                # PG 端口通不通
curl -v http://investdojo.local:8000/auth/v1/health          # HTTP 可达
```

| 症状 | 原因 | 修复 |
|---|---|---|
| `ping investdojo.local` 超时 | mDNS 抽风 | 改用 IP `192.168.1.5` |
| `ping 192.168.1.5` 超时 | 不在同一 Wi-Fi | 检查双方都在 `CMCC-e7sa-5G` |
| ping 通但 `nc :8000` 挂 | Windows 防火墙 / Docker 没跑 | Windows 上 `docker compose ps`；防火墙 `Get-NetFirewallRule -DisplayName "InvestDojo-*"` |
| HTTP 请求 401 | apikey 错 | 从 Windows `.env` 重新读 `ANON_KEY` |
| HTTP 请求 404 | URL 路径错 | 必须是 `/rest/v1/<table>` 或 `/auth/v1/<endpoint>` |

### 4.2 某个容器一直 Restarting

```powershell
cd E:\project\ownproject\invest_dojo\investdojo\infra\supabase-lite
docker compose ps                       # 看哪个在 restart
docker compose logs --tail=50 <service> # 看最后 50 行日志
```

**常见 fatal 错误对照表**：

| 日志关键字 | 含义 | 修复 |
|---|---|---|
| `API_EXTERNAL_URL missing` | GoTrue 环境变量没传 | 检查 compose 的 auth.environment |
| `parsing "": invalid syntax` | GoTrue 某数字字段空串 | 见坑表 `GOTRUE_SMTP_PORT` 条 |
| `must be owner of function` | init SQL 和 GoTrue 抢建函数 | 清 data 目录重建 |
| `plugin X not enabled` | Kong 没启用某插件 | 检查 `KONG_PLUGINS` 环境变量 |
| `pq: password authentication failed` | authenticator/supabase_auth_admin 密码没设 | 重跑 up.ps1 的第 6 步（ALTER ROLE） |

### 4.3 PostgREST 返回 401/403

| 响应 | 原因 | 修复 |
|---|---|---|
| `{"code":"PGRST301","message":"JWT expired"}` | Key 过期（10 年一次） | 重新签 Key |
| `{"code":"42501","message":"permission denied"}` | RLS 拦了 | 用 service_role key；或检查 RLS policy |
| 空数组 `[]` | 没数据 / RLS 完全屏蔽 | 先用 service_role 查，排除 RLS |

## 5. 服务调用链

```
用户浏览器
   │
   ▼
Next.js (Mac :3000)
   ├─ 页面请求 → Mac 本地路由
   ├─ Auth 请求 → http://investdojo.local:8000/auth/v1/*  → Windows Kong → GoTrue
   └─ 数据读 → 分两条路：
        a) 浏览器 → Kong → PostgREST（走 ANON_KEY，受 RLS 保护）
        b) 浏览器 → Node orch (:8787) → PostgREST（走 SERVICE_ROLE_KEY，绕过 RLS）

Python 服务 (Mac :8000-8005)
   └─ httpx → http://investdojo.local:8000/rest/v1/*  → Kong → PostgREST
        用 SERVICE_ROLE_KEY（在 .env 里）
```

**RLS 行为**：
- 前端用 ANON_KEY 调 `/rest/v1/klines_all` → 有公开读策略，能返回数据
- 前端用 ANON_KEY 调 `/rest/v1/sessions` → 需要登录，未登录返回空
- Python 服务用 SERVICE_ROLE_KEY → 全部绕过 RLS

## 6. 端口与凭据

### 6.1 端口清单

| 服务 | 监听位置 | 端口 | 对外 |
|---|---|---|---|
| Next.js web | Mac | 3000 | 仅 localhost |
| Node orchestrator | Mac | 8787 | 仅 localhost |
| data-svc | Mac | 8000 | 仅 localhost |
| feature-svc | Mac | 8001 | 仅 localhost |
| train-svc | Mac | 8002 | 仅 localhost |
| infer-svc | Mac | 8003 | 仅 localhost |
| backtest-svc | Mac | 8004 | 仅 localhost |
| monitor-svc | Mac | 8005 | 仅 localhost |
| Redis | Mac | 6379 | 仅 localhost |
| MinIO (S3 API) | Mac | 9000 | 仅 localhost |
| MinIO (Console) | Mac | 9001 | 仅 localhost |
| **Supabase Kong** | **Windows** | **8000** | **局域网** |
| **Postgres** | **Windows** | **5432** | **局域网** |

**注意端口冲突**：Mac 端 data-svc 和 Windows 端 Kong 都是 8000，但因为在不同机器上所以没事。

### 6.2 凭据清单

| 名字 | 在哪 | 长什么样 |
|---|---|---|
| POSTGRES_PASSWORD | Windows `infra/supabase-lite/.env` | 24 位随机字符串 |
| JWT_SECRET | 同上 | 64 字符 base64 |
| ANON_KEY | 同上 | eyJhbGci... (JWT, role=anon) |
| SERVICE_ROLE_KEY | 同上 | eyJhbGci... (JWT, role=service_role) |
| Cloud DB 密码（旧，兜底用） | Supabase Dashboard | 用户自己记 |

**千万不要**：
- 把 `.env` commit 进 git（`.gitignore` 已配）
- 把 SERVICE_ROLE_KEY 写进前端 `NEXT_PUBLIC_*` 变量
- 把 POSTGRES_PASSWORD 贴群里

### 6.3 前端 vs 后端的 .env

| 文件 | SUPABASE_URL | 用哪个 key |
|---|---|---|
| `investdojo/infra/.env` | `http://investdojo.local:8000` | SERVICE_ROLE_KEY |
| `investdojo/python-services/.env` | 同上 | SERVICE_ROLE_KEY |
| `investdojo/apps/server/.env` | 同上 | SERVICE_ROLE_KEY |
| `investdojo/apps/web/.env.local` | **`NEXT_PUBLIC_SUPABASE_URL`** 同上 | **`NEXT_PUBLIC_SUPABASE_ANON_KEY`** 只给 ANON_KEY |

## 7. 升级路径

### 7.1 想加 Realtime（WebSocket 订阅表变化）

```yaml
# 在 docker-compose.yml 添加服务
realtime:
  image: supabase/realtime:v2.30.34
  environment:
    DB_HOST: db
    DB_PORT: 5432
    DB_USER: supabase_admin
    DB_PASSWORD: ${POSTGRES_PASSWORD}
    DB_NAME: postgres
    JWT_SECRET: ${JWT_SECRET}
    # ...
```

再在 Kong 加一条路由 `/realtime/v1/* → realtime:4000`。约 1 小时工作量。

### 7.2 想加 Storage（替代 MinIO）

同上思路，加 `supabase/storage-api` 容器，Kong 加 `/storage/v1/*` 路由。但**项目目前用 MinIO**，没必要动。

### 7.3 想把 Python 后端从 PostgREST 改成直连 PG（方案 C+）

触发条件：
- Epic 6/7 跑起来后发现 PostgREST 性能瓶颈
- 或者想彻底去 Supabase 依赖

步骤：
1. 在 `python-services/common/` 新建 `pg_client.py`（用 `asyncpg`）
2. 保留 `supabase_client.py` 不变，让两者共存
3. 新代码用 `pg_client`，老代码慢慢迁
4. 全部迁完后删 `supabase_client.py`

详见 `2026-05-02-本地栈部署完成.md §为什么这么选` 里的方案 C+ 分析。

### 7.4 想搬到云（触发：用户 > 100 或 Windows 挂了）

**零代码改动**搬家路径：

1. 云上开一个 PG 实例（Neon / Railway / 阿里云 RDS）
2. `pg_dump` Windows 本地 → `pg_restore` 云
3. 改 `.env` 的 `SUPABASE_URL` 指向云
4. 如果云不带 PostgREST 和 GoTrue：在云上再跑个 docker-compose（把我们这套 supabase-lite 原样搬过去）
5. 如果云带（比如回 Supabase Cloud Pro）：直接用它的服务，少两个容器

**PG 数据格式完全兼容**，这是选方案 B 最大的好处。

## 8. 备选方案

这些方案**当时评估过但没选**，记录下来是为了日后决策有据可查。

### 8.1 NAS 自托管（已放弃）

原计划：买群晖 NAS → Docker + Tailscale → Supabase 全栈 → 公网 HTTPS。

**放弃原因**：
- 要买硬件（~2000 元）
- Tailscale / 公网域名配置更复杂
- 全栈 10+ 容器在 NAS 上内存吃紧

当前方案用 Windows 电脑替代 NAS，硬件成本为 0，局域网 HTTP 够用。

### 8.2 Supabase 全栈（Windows）

```
postgres + postgrest + gotrue + kong + storage +
realtime + studio + meta + analytics + vector + functions
```

**放弃原因**：
- 内存 3GB+（我们精简版 600MB）
- 坑更多（kong ACL / analytics LogFlare / vector 配置）
- 项目不用 Storage / Realtime / Analytics / Functions，白背

如果将来真的要 Realtime，单独加一个容器就行（见 §7.1）。

### 8.3 换数据库（MySQL / SQLite / MongoDB）

放弃。代码所有表 DDL、RLS、42 张表的索引、17 个分区表都是 PG-specific。换一家要改代码 2 周。

### 8.4 直接改代码用 asyncpg（方案 C+）

放弃（暂时）。前端深度绑 Supabase Auth，改动量是 4-5 天，Epic 3 开发期间不适合停摆这么久。**留给 Epic 7 前再评估**（见 §7.3）。
